Project #18

Project Title: Quantum-Inspired Spin-Based Computing Architectures

*Project Description*: The project investigates utilizing the computational
significance of the lowest-energy state of classical spin networks. This
approach presents a fresh perspective on representing computational tasks
and their realization on conventional platforms. In this project, students
will help develop a microcontroller-based network implementing a
relaxation-based dynamical Ising machine, R-DIM, a computing architecture
inspired by how networks of interacting spins evolve toward low-energy
configurations. The main goal is to build a distributed system on a network
of ESP32 microcontrollers and to explore how information should be
exchanged efficiently across the network and between the network and the
PC-based host/server. Students will investigate several communication
modes, including wired links, Wi-Fi, Bluetooth, and Zigbee. Alongside the
hardware implementation, a PC-based simulation will be developed to test
design ideas before deploying them on the MCU network. The project will
involve designing message-exchange protocols, organizing communication
between nodes and a host computer, distributing computational tasks across
the network, comparative analysis of different network connectivity
topologies, and evaluating performance trade-offs such as speed,
reliability, scalability, and energy use. This work will give students
hands-on experience with embedded systems, networking, and unconventional
computing architectures.

*Student Tasks and Responsibilities*: Students will contribute to the project
through a sequence of development steps, beginning with a basic
implementation of the R-DIM on a single microcontroller connected to a host
PC. Early tasks may include setting up the reproducible ESP32 development
environment, developing a PC-MCU communication module, developing PC-side
simulation and monitoring tools, and documenting system behavior. As the
project progresses, students will help design and implement a communication
module for MCU-to-MCU interaction, with an initial focus likely on
Zigbee-based communication. They will then build and test small device
networks with the ring topology, and evaluate how reliably and efficiently
messages are exchanged between nodes. Students will be responsible for
developing and debugging firmware, creating test cases, collecting and
analyzing performance data, and comparing embedded results with PC-based
simulations. Later tasks may include refining the message-exchange
protocol, studying communication delays and synchronization issues. Future
explorations will include investigating alternative network organizations
and task distributions across the network. Students are expected to work
carefully and independently while maintaining code, documentation, and
regular communication with the project mentor and team.

*Minimum Student Qualifications*: Because the project is in its early stages,
the barrier to entry is intentionally low, and motivated junior engineering
students are encouraged to participate. Students should have basic
programming experience in C and Python, as these languages will be used for
microcontroller firmware development, PC-based tools, testing, and data
analysis. Some familiarity with embedded systems concepts, such as
microcontrollers, GPIO, serial communication, interrupts, or basic
hardware/software interaction, will be helpful, but extensive prior
experience is not required. Relevant coursework may include introductory
programming, circuits, digital logic, embedded systems, computer
architecture, signals and systems, or computer networks, though students do
not need to have completed all of these courses. A willingness to learn new
tools, read technical documentation, debug hardware and software issues,
and work through open-ended engineering problems is more important.
Students should be comfortable working both independently and
collaboratively, maintaining organized code and notes, and asking questions
when needed. An interest in unconventional computing, distributed systems,
or low-power embedded platforms would be beneficial.

** General recommendations

Using Dev Containers (popularized by VS Code, though supported by other tools) allows you to define your entire development environment *as code* and store it directly alongside your source code in GitHub. Here is how this architecture seamlessly bridges your logistical gaps.

*** 1. The Repository Structure (The Single Source of Truth)

Your private university GitHub repository will hold both the project code and the environment blueprint. You will place a `.devcontainer` directory at the root of your repository containing two critical files:

1. ~Dockerfile~ : This script pulls the official Espressif ESP-IDF Docker
   image (ensuring the exact version of the toolchain is locked in) and
   installs any extra dependencies your lab needs.

2. ~devcontainer.json~ : This configuration file tells the IDE how to build
   the container, mounts the local source code into the container, and
   automatically installs required IDE extensions (like C/C++ IntelliSense,
   CMake tools, or specific linters) for whoever opens the project.

*** 2. The Student Workflow (Frictionless Onboarding)

When a new student joins your lab, their setup process drops to a highly
reproducible 15-minute procedure:

1. **Install Prerequisites:** The student installs Docker Desktop, VS Code,
   and Git on their machine.

2. **Clone:** They clone the private GitHub repository locally.

3. **Launch:** They open the folder in VS Code. The IDE detects the
   `.devcontainer` folder and displays a prompt to "Reopen in Container."

4. **Develop:** Docker automatically pulls the ESP-IDF image, spins up the
   container, and mounts the code. The student is instantly ready to
   compile the Ising machine firmware in a heavily standardized
   environment, completely eliminating the "it works on my machine"
   syndrome.

*** 3. Knowledge Management Integration

Because the environment setup is automated, your GitHub Wiki (or Markdown
files stored within the repository) can focus purely on engineering
knowledge rather than environment troubleshooting. You can embed your SOPs,
architectural diagrams of the Ising solver, and phase planning directly in
the repository. When a student opens the Dev Container, they have the
codebase, the exact compiler, and the documentation all in one unified
window.

*** 4. The Hardware Catch: USB Pass-through

The main logistical hurdle with Dockerized embedded development is flashing
the hardware. Because the code is compiling inside an isolated container,
it does not naturally have access to the host machine's physical USB ports
to flash the ESP32.

You will need to define a standard operating procedure for USB passthrough.
On Linux, this is a simple device flag in the `devcontainer.json`. On
Windows (WSL), students will need to use a tool like `usbipd` to attach the
ESP32 to the container. Documenting this specific step meticulously in your
GitHub repository will save you countless hours of troubleshooting.


** Sub-Project Organization and Repository Structure

The ESP-IDF framework is specifically designed to handle this exact
scenario through its *Component System* and flexible CMake build structure.
You can keep all sub-projects in one repository while strictly isolating
their development and testing.

Here is how the repository can be organized to support both modular
sub-project development and the final integrated system.

*** 1. Leverage ESP-IDF Components

In ESP-IDF, any reusable piece of code (like the communication or the Ising
computing modules) should be built as a "Component." A component is
essentially a self-contained library with its own source files, header
files, and =CMakeLists.txt= file.

By designing sub-projects as components from day one, students are guided
to decouple their code. The communication module will not know anything
about Ising spins; it will only know how to take a generic payload and send
it over the network.

*** 2. The "Multi-App" Single Repository Structure

Instead of having one single =main()= function for the entire repository, the
repository houses the core components alongside several independent
applications (sub-projects).

The repository structure could look like this:

- *=components/=* /(The shared libraries)/
  - =ising_solver/= (Math, spin dynamics)
  - =network_comm/= (Wi-Fi, ESP-NOW, MQTT wrappers)
  - =hal_config/= (Pin definitions, hardware abstraction)
- *=apps/=* /(The executable sub-projects)/
  - =test_comm_module/= (A standalone =main= that only boots networking)
  - =test_solver_module/= (A standalone =main= that only runs math benchmarks)
  - =integration_main/= (The actual overarching project that links everything)
- *=.devcontainer/=* (Docker environment configuration)
- *=docs/=* (SOPs and Wiki)

*** 3. Workflow for Sub-Project Development

When a student is assigned to work entirely on the communication module,
they do not need to touch the Ising solver. Their workflow becomes highly
focused:

1. They modify code inside =components/network_comm/=.
   
2. They compile and flash only the =apps/test_comm_module/= application to
   verify their work.

3. Because =test_comm_module/= only links the networking component,
   compilation is much faster, and debugging is isolated from the rest of
   the system.

Once the communication module is stable and validated, integrating it into
the overarching project is trivial. You simply add =network_comm= to the
CMake requirements of =apps/integration_main/=, and the whole system can now
use the new networking features.

*** 4. Git Branching for Sub-Projects

To further protect the main project, enforce a Git workflow that aligns
with this structure.

- Have students create branches named after their specific tasks (e.g.,
  =feature/esp-now-comm=).

- They do their isolated development and testing in that branch.

- When they submit a Pull Request, you (or a senior student) review it to
  ensure they haven't accidentally hardcoded anything that breaks the
  separation of concerns before merging it into the =main= branch.


** PlantUML

*** PlantUML resources

+ The Hitchhiker's Guide to PlantUML: (Available at
  crashedmind.github.io/PlantUMLHitchhikersGuide/). This is an exceptional,
  community-driven guide that goes far beyond the basic syntax. It includes
  practical examples of layout tricks, real-world networking diagrams, and
  architectural mapping.
  
+ PlantText (The Web Sandbox): (planttext.com).
  Before configuring anything locally, use this online editor. It gives you
  a split-screen view: text on the left, instantly updated SVG/PNG on the
  right. It is perfect for spending an hour just memorizing basic
  syntax.

+ The Official PlantUML Language Reference (PDF): (plantuml.com).
  The official documentation is uniquely comprehensive. Instead of reading
  it cover-to-cover, keep the specific PDFs open for the diagram types you
  need (specifically the Sequence Diagram and State Diagram guides).
